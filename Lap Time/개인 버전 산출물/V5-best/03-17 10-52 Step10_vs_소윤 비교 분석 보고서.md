# Step10 vs 소윤 Ensemble 비교 분석 보고서

## 요약
- 현재 내부 의사결정에서 가장 중요한 비교는 `submission_관주v5_best.csv`와 `ensemble_v4_submission_소윤1등.csv`다.
- 리더보드 차이는 매우 작다.
  - 우리 `submission_관주v5_best.csv`: `0.7421348647`
  - 소윤 `ensemble_v4_submission_소윤.csv`: `0.7421795958`
  - 차이: `0.0000447311`
  - 현재 1등: `0.74222`
- 결론부터 말하면, 승부는 `새로운 내부 GBDT 파생`보다 `rank 구조`와 `노이즈가 적은 기본 피처셋`에서 났다.
- 다만 사전 가정과 달리, `submission_관주v5_best.csv`는 `v5_rank.csv`와는 완전히 동일하지만 `vv7_rank3_equal.csv`, `vv8_rank_eq_CX_V31_CL_XGB_CX_LGB.csv`와는 동일하지 않았다.
- 따라서 이번 보고서는 **초기 가설이 아니라 실제 파일 검증 결과**를 기준으로 정리한다.

## 2. 정체 확인: 각 제출 파일이 실제로 무엇인가

### 2-1. `submission_관주v5_best.csv`

실제 파일 비교 결과:

| 비교 대상 | 동일 여부 | 최대 절대 차이 |
|---|---|---:|
| `v5_rank.csv` | 동일 | 0 |
| `vv7_rank3_equal.csv` | 동일 아님 | 0.0227835297 |
| `vv8_rank_eq_CX_V31_CL_XGB_CX_LGB.csv` | 동일 아님 | 0.0227835297 |

핵심 해석:
- `submission_관주v5_best.csv`는 **정확히 `v5_rank.csv`와 동일**하다.
- 하지만 **저장된 이전 equal-rank trio 파일인 `vv7_rank3_equal.csv`, `vv8_rank_eq_CX_V31_CL_XGB_CX_LGB.csv`와는 동일하지 않다.**
- 즉 이 제출은 `step_10` 내부에서 생성된 **best rank 결과**이며, 단순히 예전 equal-rank trio 파일을 다시 낸 것은 아니다.

추가 관찰:
- 기존 저장 후보 중 가장 가까운 파일은 `vv8_rank_opt_CX_V31_CL_XGB_CX_LGB.csv`와 `vv8_div_best_opt.csv`였다.
- 따라서 `submission_관주v5_best.csv`는 **equal-rank trio와 완전히 같은 계보라기보다, trio 계열의 rank 최적화 쪽에 더 가깝다**고 보는 편이 정확하다.

### 3-1. Step 10 계열

| 항목 | 내용 |
|---|---|
| 코드 | `step_10_codex_cat.py` |
| 목적 | 새 CatBoost를 학습하고 기존 strong OOF와 blend 탐색 |

| bagging_temperature | 0.8 |
| rsm | 0.8 |
| border_count | 128 |
| early stopping | 코드상 AUC 기준 CatBoost 학습 루프 |

#### Step 10 피처 구조

| 피처군 | 내용 |
|---|---|
| missing indicators | 결측 지표 추가 |
| procedure decomposition | 시술 유형 분해 |
| egg donor feature | 난자 기증 관련 feature |

| Codex IVF/DI split | numeric `12개` |

해석:
- step_10은 **새 CatBoost 자체를 키우는 실험**이라기보다, 새 CatBoost를 strong anchor들과 함께 넣고 blend 최적화를 찾는 구조였다.
- 따라서 `step_10` 결과는 새 Cat 단독 성능보다도 **최종 blend 선택**을 어떻게 했는지가 더 중요하다.

### 3-2. 소윤 ensemble_v4 계열

| 모델 | OOF AUC | 비고 |
|---|---:|---|
| `xgb_v2_reg_relax` | 0.740113 | reg_relax |
| `catboost_v2` | 0.740053 | baseline v2 |
| `lightgbm_v1` | 0.739865 | baseline v1 |
| final weighted rank | 0.740444 | `0.33 / 0.41 / 0.26` |

핵심 해석:
- 개별 모델 OOF는 아주 높지 않다.
- 그런데도 final LB가 높게 나온 이유는
  - simple rank
  - cat 비중 0.41
  - 107개 base 중심의 low-noise generalization
  조합이 잘 맞았기 때문이다.

### 3-3. threshold CSV 해석

확인 결과:

| 모델 | best_f1 threshold | best_accuracy threshold |
|---|---:|---:|
| XGB | 0.3 | 0.5 |
| CatBoost | 0.3 | 0.5 |
| LightGBM | 0.3 | 0.5 |

해석:
- threshold curve 자체는 세 모델이 거의 비슷하다.
- 이 대회는 AUC 기준이므로, threshold 분석 CSV는 **보조 자료**일 뿐이다.
- 모델 선택의 핵심 근거는
  - OOF
  - LB
  - 앙상블 구조
  - 피처 노이즈 수준
  이어야 한다.

## 4. 왜 소윤 rank가 우리 v5보다 약간 더 높았나

### 핵심 수치

| 항목 | 값 |
|---|---:|
| 우리 v5 LB | 0.7421348647 |
| 소윤 LB | 0.7421795958 |
| 차이 | 0.0000447311 |
| 우리 trio rank OOF 계열 | 약 0.740875 |
| 소윤 final weighted rank OOF | 0.740444 |

### 원인 분석

| 원인 | 설명 |
|---|---|
| 1. weighted rank가 equal rank보다 소폭 더 맞았음 | 소윤 제출은 `0.33 / 0.41 / 0.26` weighted rank이며, 현재 public에서는 이 미세한 weighting이 더 잘 맞았다. |
| 2. `107개 base` 중심이라 noisy codex additions가 적었음 | 소윤 branch는 비교적 적은 피처와 단순한 파이프라인이라 public 일반화가 유리했다. |
| 3. Cat 비중 `0.41`이 LB 친화적이었음 | 세 모델 중 CatBoost 비중을 가장 크게 둔 것이 public에 더 잘 맞았다. |
| 4. step_10의 새 CatBoost가 최종 우위로 확정되진 못했음 | 우리 제출은 `step_10`에서 생성된 rank 결과지만, 이전 trio/equal 계열보다 명확히 우월하다고 보기엔 근거가 부족했다. |

### 냉정한 결론
- 우리 v5가 약한 파일은 아니다.
- 하지만 `step_10`의 새 CatBoost가 **public에서 소윤 weighted-rank를 넘어서는 새 축을 만들었다고 보긴 어렵다.**
- 실제 차이는 매우 작고, 그 차이는 `새 feature 발명`보다 `rank 구조`와 `노이즈 관리`에서 났다.

## 최종 결론
- 소윤 2등 파일은 정확히 `xgb_v2_reg_relax + catboost_v2 + lightgbm_v1`의 `0.33 / 0.41 / 0.26 weighted rank`다.
- 우리 `submission_관주v5_best.csv`는 `v5_rank.csv`와는 완전히 동일하지만, 사전 가정과 달리 `vv7_rank3_equal.csv`, `vv8_rank_eq_CX_V31_CL_XGB_CX_LGB.csv`와 동일하지 않았다.
- 따라서 지금까지의 검증을 기준으로 보면:
  - **public에서 더 잘 먹힌 것은 simple/weighted rank 구조**
  - **새로운 내부 GBDT 파생 자체는 결정적 우위를 만들지 못함**
  - **피처는 `107 base` low-noise branch와 `115개 CatBoost anchor` 두 축으로 정리하는 것이 가장 합리적**
- 오늘/내일 전략은 `새 모델 여러 개`가 아니라
  - `소윤 strong rank`
  - `우리 strong rank`
  - 둘의 simple cross-rank
  를 빠르게 검증하는 쪽이 맞다.
